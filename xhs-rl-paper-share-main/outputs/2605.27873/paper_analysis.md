# AIBuildAI-2 深读

## 核心结论

AIBuildAI-2 的关键贡献不是新模型结构，而是把自动 ML 工程 Agent 接到一套分层、按需读取、任务后可写回的知识系统。它试图把零散在竞赛复盘、技术博客、官方文档和历史实验目录中的工程经验，变成可寻址的长期资产。

## 方法链条

系统由 Setup、Manager、Designer、Coder、Tuner 和 Aggregator 组成。Manager 同时维护 7 个 solution repositories，根据设计、代码、日志和验证结果继续投入强路线并淘汰弱路线。知识系统包含约 30 类 L1 高层技能指南和约 1000 份 L2 细粒度文档。每次调用先看到 L1 类别目录，再执行 `query(k)=L1[k]`；只有需要具体配方时才执行 `query(k,j)=L2[k][j]`。

任务结束后，L2 Builder 从完整工作目录提炼成功策略、失败模式与可复用 takeaway，L1 Builder 再把新增经验压缩进高层指南。新增网页资料也走同一写入管线。因此系统进化的是知识内容，而不是底模参数、提示词或 Agent 编排。

## 证据

MLE-Bench 75 项任务的总体奖牌率为 70.7%，高、中、低难度组分别为 46.67%、73.68%、81.82%。心脏病竞赛中，系统自动形成 CatBoost、XGBoost、LightGBM 的 rank-average 融合，私榜 AUC 0.95529，排名 288/4371。OpenADMET 中，系统组合图消息传递网络、分子指纹/描述符 LightGBM 与逐任务加权融合，MA-RAE 0.6300，排名 40/103。

## 证据边界

论文没有关闭 L1、L2、动态加载或经验写回的组件消融，也没有展示知识库随任务数量增长带来的纵向迁移曲线。同设置真实竞赛比较能够说明完整系统优于 AIBuildAI 与 MLEvolve，但架构差异不止一个变量，不能据此量化每个知识组件的独立贡献。

复现成本也不低：Claude Opus 4.7、temperature 1.0、每任务 7 条并行路线、24 小时、24 vCPU、256 GB RAM 和一张 A100。知识库的完整内容、构建版本与更新历史同样会影响结果。

## 可迁移启示

生产级 Agent 的长期记忆不应只是保存聊天记录。更可靠的做法是：目录常驻、正文按需加载、任务结束后离线提炼，并为知识条目维护来源、版本、置信度、冲突检测和回滚。AIBuildAI-2 很好地展示了这个方向，但还没有证明系统会随着长期运行稳定地越来越强。

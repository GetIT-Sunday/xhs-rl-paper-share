# 让AI自己积累建模经验

AIBuildAI-2 想解决的，不是让大模型再多想几步，而是让自动建模 Agent 真正拥有可积累的工程知识。

为什么这件事重要？端到端建模包含模型选择、特征处理、训练排错、交叉验证、调参与融合。大量决定依赖散落在竞赛复盘、技术博客、库文档和代码仓库里的经验，通用 LLM 的参数知识很难始终覆盖。

**核心痛点**

1. 只靠参数知识：静态、会过时，而且工程细节稀疏。
2. 临时网络搜索：信息新，却噪声高、查询结果不稳定，任务结束后也不会自动沉淀。
3. 固定文档库：比搜索可靠，但不能吸收新资料和系统自己的实验经验。

**有意思的设计**

1. 系统把知识分成两层：约 30 类 L1 高层指南负责导航，约 1000 份 L2 文档保存具体配方和失败模式。
2. Agent 先调用 `query(k)` 读取某类指南及文档目录；只有高层信息不够时，才调用 `query(k,j)` 加载具体文档。重点不是知识越多越好，而是让当前决策只看到相关知识。
3. Manager 同时维护 7 条方案路线，Designer、Coder、Tuner 反复迭代，Aggregator 最后选择或融合最强方案。
4. 每次任务结束后，L2 Builder 从完整实验目录提炼 takeaway，L1 Builder 再更新高层指南；新竞赛复盘与库更新也进入同一管线。

**实验怎么看**

在 75 项 MLE-Bench 上，AIBuildAI-2 的奖牌率为 70.7%；高、中、低难度组分别为 46.67%、73.68% 和 81.82%。心脏病预测竞赛中，系统自动构造 CatBoost、XGBoost、LightGBM 的排序融合，私榜 AUC 0.95529，排名 288/4371。OpenADMET 盲测中，它组合图网络、分子指纹和 LightGBM，MA-RAE 为 0.6300，排名 40/103。

**我的判断**

这篇论文的工程方向很扎实：长期记忆不该只是保存对话，而应成为分层、可检索、可审计的技能资产。但证据也有明显边界。论文没有关闭 L1、L2、动态加载或经验写回的组件 ablation，也没有展示知识库随任务数量增长的纵向学习曲线。同设置竞赛对比能说明完整系统很强，却不能量化每个知识机制的独立贡献。

复现成本同样不低：Claude Opus 4.7、7 条并行路线、每任务 24 小时、24 vCPU、256 GB RAM 和一张 A100。作者也承认，系统更新的是知识，Agent 的 prompts、行为代码和编排 harness 仍然固定。
